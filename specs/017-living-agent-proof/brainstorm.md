# Living Agent Proof - Brainstorm

**Stage:** approved by owner instruction to run all phases
**Domain:** Complex
**Date:** 2026-07-21

## Press release

DESIGN:OS can now prove when a design agent learns, applies that learning, and improves a
workflow. The proof does not rely on event counts or a self-awarded score. It traces user or
project evidence into a retrieved lesson, a changed design decision, a later artifact, and a
measured outcome.

Teams can inspect four distinct levels: the loop exists, evidence became learning, later work
applied it, and comparable outcomes improved. Unsupported claims remain visibly unproven.

## User stories

- As a DESIGN:OS user, I want proof that prior feedback affects later work so I do not repeat it.
- As a design lead, I want every claimed memory application linked to an artifact decision.
- As a maintainer, I want escaped defects converted into executable prevention checks.
- As an evaluator, I want controlled comparisons so improvement is not inferred from chronology.
- As a product owner, I want project isolation and contradiction tests to prevent stale leakage.

## MoSCoW

| Priority | Item | Rationale |
|---|---|---|
| Must | Causal applied-memory receipts | Distinguishes stored memory from changed behavior |
| Must | Defect-learning records with negative fixtures | Proves repeated-defect prevention |

| Must | Three dogfood fixtures | Grounds the contract in real project evidence |
| Must | Nested-store and stale-heartbeat diagnostics | Existing projects already exhibit these failures |

| Should | Contradiction and isolation evidence | Prevents stale or cross-project learning |

| Won't | Auto-promote project lessons globally | Human-reviewed librarian remains the trust boundary |

## Appetite and no-gos

**Appetite:** one full feature cycle. Ship the deterministic proof contract, real fixtures,
diagnostics, and reports. Leave model-run automation for a later layer.

**No-gos:**

- No score derived from the number of memory events.
- No claim that a later result improved merely because it occurred later.
- No mutation of VSF-PCP or platform-design-system.
- No global promotion without repeated evidence or explicit approval.

## Research protocol

| Step | Finding |
|---|---|
| Benchmark | VSF-PCP proves persistent identity; platform-design-system proves escaped defects became gates |
| First principles | Learning is a causal change in later behavior, not stored text |
| Framework | Counterfactual evaluation plus traceability receipts and executable regression tests |
| Cross-domain | Safety engineering converts incidents into controls and recurrence monitoring |
| Trade-off | Strong proof is slower and requires comparable tasks; honesty is worth the cost |
| Executability | Deterministic validation reads evidence files; host agents create evidence during work |

## Product decision

The proof ladder is ordered and non-substitutable:

1. **ALIVE** - learning infrastructure exists and has a learning signal.
2. **LEARNING** - source evidence has become a bounded lesson or prevention rule.
3. **APPLIED** - a later decision cites and uses that lesson.
4. **IMPROVING** - a controlled or longitudinal comparison clears declared thresholds.

A project receives the highest level it can fully support. Missing evidence is reported, never
silently downgraded into a pass.
