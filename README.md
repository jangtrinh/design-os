# ease-design

A multi-runtime **design-cli**. Describe *what* you want in plain words; ease-design
supplies all of *how it looks* ‚Äî production-grade UI with the taste of a pro designer,
generated by the agent CLI you already use (Claude Code, Codex CLI, Antigravity CLI).

> Distilled from [EaseUI](../EaseUI)'s design engine. The binary and slash-command
> namespace are `ui` / `/ui:*`.

**‚Üí New here? [QUICKSTART.md](QUICKSTART.md) gets you to a first design in ~60 seconds.**
See real, gate-clean output in the proof gallery:
`examples/generated/live-2026-05-30/index.html`.

## Who it's for

ease-design serves two audiences from one engine:

- **Designers** ‚Äî turn intent into polished, on-system UI without hand-writing HTML/CSS.
  You work entirely through plain-language `/ui:*` commands in an agent CLI: describe what
  you want, get production HTML out, quality enforced by a taste gate you don't have to
  babysit. No design tokens to hand-edit, no config files, no taste vocabulary to learn.
- **Developers** ‚Äî the same `/ui:*` commands, plus a deterministic `ui` binary you can
  script directly. Token-bound output, a component registry, and a machine-checked quality
  floor ‚Äî not a black box.

The agent CLI is the interface for both. Both talk to the same two sources of truth, so a
designer's output and a developer's output obey the identical design system ‚Äî one command
surface, one engine, one quality bar.

## How it works

ease-design ships two runtime-neutral sources of truth and a thin per-runtime adapter:

- **`knowledge/`** ‚Äî a plain-Markdown knowledge core (taste rubric, persona library,
  generation rules, component catalog, color science) the host model reads directly.
- **`ui` binary** ‚Äî a deterministic Node CLI for all non-LLM work (autofix, layout
  validation, taste linting, token compilation, color math, component registry, export).
  No network, no model calls ‚Äî same inputs, same bytes.
- **Adapters** ‚Äî `ui init` generates the config a given agent CLI needs to know *when*
  to read the knowledge core and *when* to shell out to `ui`. The host CLI's own model
  generates the HTML ‚Äî **no API keys to manage**.

A hard-gate critique‚Üírefine loop scores every generation across 6 craft axes + 1
consistency axis. The model scores the subjective axes; a deterministic `ui taste-lint`
floor catches the machine-checkable rule breaches underneath ‚Äî so quality is *enforced*,
not merely suggested.

## Requirements

## Install

```sh
git clone <repo-url> ease-design
cd ease-design
npm install
npm run build
npm link          # exposes the `ui` binary on your PATH
```

The package is publish-ready and **install-verified**: `npm pack` ‚Üí install into a clean
project boots the `ui` binary, and `ui doctor` confirms the bundled `knowledge/`, `schemas/`,
and `templates/` resolve from inside `node_modules/ease-design/`. Once published,
`npx ease-design` / a global install replaces the clone-and-link step. Run `ui doctor` after
install to confirm a healthy setup.

### Releasing (maintainer)

Publishing is automated and guarded ‚Äî no manual `npm publish`:

```sh
npm version patch|minor|major   # bumps package.json + tags
git push --follow-tags          # triggers .github/workflows/release.yml
```

The release workflow re-runs all four gates, verifies the git tag matches `package.json`
version, and only then `npm publish`es (with provenance). A `prepublishOnly` script runs the
gates locally too, so a broken build can't be published by any path. One-time setup: add an
`NPM_TOKEN` repo secret.

## Usage

```sh
ui guide                     # plain-language map of what you can do (start here)
ui doctor                    # verify the install is healthy
ui init --runtime claude     # wire ease-design into the current project
ui doctor --cwd .            # confirm the project wired up correctly
```

**New to ease-design? Run `ui guide`.** It's the designer's on-ramp: an
intent-organized, jargon-free map of the `/ui:*` workflow ("start a design",
"tweak it", "match a screenshot", ‚Ä¶) ‚Äî so you know what to type without reading
14 engine commands. `ui --help` points there too.

`ui doctor` is the first-run smoke test: it checks Node ‚â• 20, that the bundled
`templates/` and `knowledge/` resolve with their key files, and ‚Äî with `--cwd` ‚Äî
that a project's adapter manifest points at a knowledge core that actually exists.
No more silent setup failures.

`ui init` writes a per-runtime adapter tree. For Claude that is **10 slash-commands**
(`.claude/commands/ui/`) + **7 skills** (`.claude/skills/ease-design-*/`); Antigravity
gets the equivalent workflow tree; Codex gets an `AGENTS.md` block. Each generated
wrapper points at the runtime-neutral templates and anchors the absolute `knowledge/`
path, so the workflows resolve no matter where the project lives.

In an agent CLI the workflows surface as `/ui:generate`, `/ui:iterate`, `/ui:refine`,
`/ui:redesign`, `/ui:extract`, `/ui:from-ref`, `/ui:from-url`, `/ui:figma`,
`/ui:slides` (Claude Code namespace).

`/ui:from-url <url>` extracts a live site's design system into `./<slug>/` ‚Äî a
self-contained folder with `DESIGN.md` (spec), `DESIGN.preview.html`, the raw
`source.html` + `source.css` audit trail, `tokens.json` (frequency-ranked source tokens
with provenance), `run-summary.md`, and an `audit.md` whose exit code gates the workflow
(5 audit families: format, source-fidelity, ref-integrity, accessibility, discipline).
Flags: `--name <slug>`, `--out-dir <path>`, `--force`. (The deterministic half of this
flow is the `ui designmd extract-tokens / snapshot / audit` subcommands; the workflow
orchestrates them.)

## The happy path

The core flow is built so someone with no design vocabulary can produce
production-quality UI on demand:

1. **Describe the intent in plain words.** `/ui:generate landing page for a new gym`.
   No mood-board, no color picker, no font list.
2. **ease-design picks personas + compiles a design system.** The host model reads
   `knowledge/persona-index.md`, scores the intent against 23 curated personas, and
   selects a diverse top-K. Then `ui ds init` compiles a project-scoped DS ‚Äî semantic
   tokens, a component registry, and a hash-sealed manifest ‚Äî from the chosen persona.
   The tokens are compiled to a Tailwind `@theme` block the generated HTML consumes as
   utilities, so "use the design system" is mechanical, not retyped hex.
3. **Variants come back.** Each draws from a different selected persona, so you pick by
   eye rather than by vocabulary. The critique gate (model scoring + the `ui taste-lint`
   floor) scores each before you see it; failing variants regenerate.
4. **Refine in plain words.** `/ui:iterate make the hero warmer, tighten the headline`.
   The host model translates that into token / layout edits; a surgical line-diff applies
   it where it can; the critique gate re-scores; the DS hash-seal stays intact.

No keys to manage, no design-system spec to write, no taste vocabulary to learn.

## Status

- **`knowledge/` core:** a 6+1-axis taste rubric, 23 personas across 7 families, 32
  components across 8 categories, color science, token taxonomy, prompt modes.
- **`ui` binary:** 14 commands ‚Äî color math, token compilation, autofix, layout
  validation, **taste linting**, **install health (`doctor`)**, component registry,
  edit strategy, JSON streaming, standalone export, manifest + adapter generation,
  design-system SSOT (`ds init / context / change-token / status`), and the `designmd`
  extractor family.
- **Workflows:** 10 host-model workflows (`generate`, `iterate`, `refine`, `redesign`,
  `extract`, `from-ref`, `from-url`, `figma`, `slides`, `critique`) + 7 skills, both
  adapter-generated per runtime by `ui init`.
- **Critique gate:** a hard pass/fail loop. The model scores the subjective axes; the
  deterministic `ui taste-lint` floor enforces the machine-checkable rules (body ‚â• 16px,
  on-grid spacing, one icon family, tinted shadows, non-linear easing, token-bound
  colors) so an axis with a real violation cannot pass.
- **749 unit tests passing**; zero runtime dependencies (devDeps only); four CI gates
  (typecheck, lint, build, test) green.

  failure mode (the model passing output that breaks a hard rule).

## Repo layout

| Path | Purpose |
|---|---|
| `knowledge/` | Markdown knowledge core ‚Äî taste rubric, personas, components, color science, token taxonomy, prompt modes |

| `src/commands/` | one file per `ui` subcommand |
| `src/core/` | shared deterministic logic ‚Äî color math, tokens, registry, autofix, layout + taste linting, validation |
| `src/adapters/` | per-runtime adapter generation for `ui init` |
| `schemas/` | JSON Schemas ‚Äî design tokens, component registry, DS manifest |
| `templates/` | workflow + skill Markdown templates (source of truth for adapter generation) |
| `examples/` | synthetic walkthrough + (post-dogfood) generation outputs |
| `docs/` | audit-gate report, journals |
