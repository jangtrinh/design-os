# Dogfood Pass тАФ 2026-05-24

End-to-end exercise of every `ui` subcommand and every adapter file produced by
`ui init`, run against the freshly built `dist/cli.js`. Two real defects found
and fixed; one heuristic limitation logged.

## Defects fixed

### D1 тАФ Stdin reading crashed in the bundled binary

**Symptom:** Any command accepting `-` as a file argument
(`registry register --markup -`, `strip-fences -`, `parse-json-stream -`,
`edit-strategy apply --diff -`) failed with:

    ui: cannot read markup from stdin: Dynamic require of "fs" is not supported

Vitest masked the bug because vitest imports source directly; the bug only
appears when tsup bundles `src/cli.ts` as ESM and the runtime hits
`require("node:fs")`.

**Fix:** `src/core/stdin-reader.ts` тАФ replaced deferred `require()` calls with
top-level `import` statements. Import-time side effects are fine: stdin is only
read when the returned function is called.

**Regression test:** `tests/cmd-stdin-built-binary.test.ts` тАФ spawns the built
binary with piped input for each of the four stdin-consuming commands.

### D2 тАФ `ds context` dropped semantic tokens after `change-token`

**Symptom:** After `ui ds change-token color.primary --value "#FF0066"`, the
token disappeared from the host-model context block, leaving the model with
no semantic primary color to use.

**Root cause:** `semanticTokens()` in `src/core/ds-context.ts` discriminated
semantic-vs-primitive by `isAlias($value)`. `change-token` converts an alias
to a literal hex, so the filter silently dropped the now-literal semantic.

**Fix:** Mark semantic tokens at expand time with
`$extensions.ease.layer = "semantic"` (`src/core/persona-expand.ts`). The
context filter now keys on the marker first, falling back to alias detection.
`ds change-token` preserves `$extensions` via shallow-spread, so the marker
survives the mutation.

**Regression test:** `tests/cmd-ds-context.test.ts` тАФ asserts `color.primary`
remains in the context block after `change-token` converts it to a literal.

## Heuristic limitations (documented, not bugs)

### H1 тАФ `absolute-without-relative` matches text content

`validate-layout`'s rule scans the whole document for the literal substrings
`absolute` and `relative`. Inline text content can trigger false negatives
(e.g. a paragraph containing the word "relative" suppresses the warning).
Documented in the rule's `--help` text as heuristic. No fix attempted тАФ
parsing class attributes would change the rule's scope and isn't required
for v1.

### H2 тАФ `cdn-urls` autofix is unpkg-specific

The `--help` text says "Replace versioned Lucide CDN URLs with @latest" but
the regex only matches `unpkg.com/lucide@...`. jsDelivr and other CDNs are
left untouched. Behavior matches the rule comment in source; help text could
be more specific in a future doc pass.

## Adapter validation

`ui init --all` produces 31 files across the three runtimes. All checked:

- YAML frontmatter present on every wrapper (`init.md` wrappers are
  binary-execution stubs and intentionally have no template path)
- Every workflow/skill wrapper's embedded `` `templates/.../*.md` `` path
  resolves to a real file on disk
- Every `invoke skill `тАж`` reference in a workflow wrapper maps to an
  installed `SKILL.md` directory in the same runtime tree
- Source templates' cross-refs to other templates and `knowledge/*.md` all
  resolve

## What this dogfood pass does *not* cover

The host-model workflows (`/ui:generate`, `/ui:iterate`, `/ui:extract`,
`/ui:from-ref`, `/ui:slides`) are Markdown prompts read by a host AI runtime.
They were verified as **valid wrappers pointing at valid templates with
valid skill references**, but actually running them тАФ choosing personas,
generating HTML variants, scoring with the taste rubric тАФ requires a host
CLI session with a model. That remains the live dogfood gap noted in v1
ship docs.
